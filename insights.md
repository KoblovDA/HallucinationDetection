## [2026-05-21] Размер и фильтрация ToolACE

- **Контекст:** загрузили `Team-ACE/ToolACE` (один `data.json`, 35 MB) и посмотрели, сколько диалогов реально пригодны для нашей задачи (нужна цепочка user → tool_call → tool_output → финальный текст ассистента).
- **Что наблюдали:**
  - Всего диалогов: **11 300**
  - С хотя бы одним tool-turn: **797** (≈ 7%)
  - Чистых триплетов «user / tool_output / assistant_text»: **741** (строгое сопоставление подряд)
  - Слабых пар «tool_output → assistant_text» (с любым предшествующим user): **1 034**
- **Гипотеза почему:** ToolACE преимущественно учит модель *звать* тулы, а не отвечать после их исполнения. Tool-execute + natural language follow-up — редкий паттерн.
- **Что делать дальше:** работаем с 1 034 триплетами как с базой. Это даёт ~1 000 примеров на каждый из 3 типов галлюцинаций. Для тренировки на ToolACE придётся либо аугментировать (несколько вариантов порчи на один сэмпл), либо подмешать данных из RAGTruth для тренировки encoder-детектора.

- **Что делать дальше:** rule-based Type 1 работает на ≈77% значений. Для 23% либо пропускаем, либо подключаем LLM-перефразировку. Этого хватит, чтобы инжектить почти каждый сэмпл — нужен 1 валидный кандидат на сэмпл, не все.

## [2026-05-21] Загрязнённые пулы значений по имени поля

- **Контекст:** при наивной агрегации значений по последнему ключу — `condition` собирает значения из ВСЕХ доменов сразу.
- **Что наблюдали:**
  - `condition`: `{Used-Like-New, Used-Acceptable, Used-Good, Partly Cloudy, New}` — смесь товаров и погоды.
  - Тупой случайный swap → ответ «погода Used-Like-New», что выглядит как очевидная ерунда и обучит детектор тривиальному паттерну.

- **Что делать дальше:** два варианта:

  2. кластеризация значений внутри поля по эмбеддингам (отделить «погодные» от «товарных»).
  Для длинного хвоста уникальных значений (5–10k) — опционально однократный LLM-проход для генерации альтернатив с кэшированием.

## [2026-05-21] Главный механизм rule-based Type 1: list-swap внутри одного tool output

- **Контекст:** изначальное предположение было, что для rule-based Type 1 нужен «пул альтернатив» из истории датасета (cross-sample). Но 94% тулов уникальны (87.3% от инвокаций — одноразки), значит cross-sample пул почти всегда пуст.
- **Что наблюдали:** декомпозиция 83.4% покрытия `(tool, field)` пула:
  - **76.3% — из того же сэмпла** (in-sample): tool output содержит список объектов, и поле повторяется с разными значениями.
  - 1.0% — только cross-sample (повторяющиеся тулы).
  - 6.0% — обе стратегии дают альтернативы.
  - 16.6% — нет пула вообще.
- **Гипотеза почему:** ToolACE содержит много диалогов с API, возвращающими списки (top-N, listings, search results). 220 из 1034 триплетов (21%) имеют рич-листы из ≥3 объектов. Внутри такого списка значения одного поля для разных сущностей взаимозаменяемы (altitude самолёта 101 ≠ altitude самолёта 102, но обе валидные альтитюды).
- **Почему это особенно ценная стратегия:**
  - Подменённое значение находится в самом tool output → детектор не может «искать неправдоподобные» (все цифры реальные). Детектор обязан понимать **к какой именно сущности относится значение**.
  - Это реалистичный тип галлюцинаций — реальные LLM именно так путают атрибуты сущностей в списках. У RAGTruth это категория «Subtle Conflict».
  - Спаны точные (substring swap).
- **Что делать дальше:** in-sample list-swap — **главная стратегия** инжекции Type 1. Cross-sample — лишь fallback для скаляров повторяющихся тулов. Также type-based замена (число/дата/bool/URL) ортогональна и работает на тех же сэмплах как альтернативный источник кандидатов.

## [2026-05-21] Конечное покрытие Type 1 rule-based инжекцией

- **Контекст:** интересует не «доля значений», а «доля сэмплов, в которых найдётся хоть один валидный кандидат на инжекцию».
- **Что наблюдали:** из 1034 триплетов с применением иерархии стратегий (type-based для чисел/дат/bool/URL, in-sample pool для списков, cross-sample pool для повторяющихся тулов):
  - **75.2% сэмплов** имеют ≥1 кандидат.
  - **55.3% сэмплов** имеют ≥3 кандидата (можно выбирать наиболее контрастный).
  - 24.8% сэмплов — пусто (ассистент не цитирует значения, либо все значения — осиротевшие скаляры).
- **Гипотеза почему:** оставшиеся 25% — это короткие ответы-резюме без цитирования конкретных полей, либо сэмплы с уникальными скалярными выходами (одна цифра/строка из одноразового тула).
- **Что делать дальше:** реализуем pure rule-based на ~780 сэмплов. Этого с запасом хватает для train/val/test split. Опционально потом подключим LLM для оставшихся 25% (мини-промпт «придумай противоречащую альтернативу для X в контексте Y», substring swap → точные спаны сохраняются).

## [2026-05-21] Bool в ответе почти всегда перефразирован — substring swap не работает

- **Контекст:** проверка работоспособности type-based bool-инверсии для Type 1.
- **Что наблюдали:** 173 bool-значения в tool_out:
  - Дословно как `"true"`/`"false"` в ответе: **5.2%**.
  - Перефразированы: **94.8%** (`is_correct: true` → `"(Correct)"`, `exists: true` → `"does indeed exist"`, `isValid: true` → `"is valid"`, `success: false` → `"failed"`).
- **Гипотеза почему:** LLM-ассистенты переводят булевы значения в естественные английские конструкции, зависящие от имени поля. Делать общее правило невозможно — конкретная фраза варьируется (`"Correct"`, `"valid"`, `"available"`, `"successful"`, …).
- **Что делать дальше:** **bool ИСКЛЮЧАЕМ из rule-based Type 1**. Его доля — 1.3% leaf values, ROI плохой. Оставляем для LLM-fallback (мини-промпт «переверни смысл этой фразы»).

## [2026-05-21] Даты переформатируются по предсказуемому набору паттернов

- **Контекст:** проверка, можно ли надёжно работать с датами type-based методом, учитывая что ассистент часто переводит ISO в natural language.
- **Что наблюдали:** 553 ISO-даты в датасете:
  - Verbatim или ISO prefix без времени: **54.4%**.
  - Поймали reformat-паттерном: **38.0%**.
  - Не упомянуты в ответе: 7.6%.
  - **Итого findable: 92.4%**.
- **Распределение паттернов** (когда verbatim не сработал):
  - `Month D, YYYY` (e.g., `January 15, 2024`): 130 случаев — основной.
  - `Month D` без года (`May 20`): 55.
  - `ISO date prefix` (без времени): 13.
  - `Month YYYY` (только месяц/год): 13.
  - `Month Dth, YYYY` с ordinal'ом: 9.
  - `Dth Month YYYY`: 3.
- **Гипотеза почему:** американский стиль натуральных дат доминирует; ordinal-варианты редки, но встречаются. Набор замкнутый.
- **Что делать дальше:** в type-based date swap'е перебираем эти 6-7 паттернов: для каждой ISO-даты пытаемся найти её в ответе сначала verbatim, потом в каждом из этих форматов. Если нашли — генерим новую дату (сдвиг на random 7-365 дней) **в том же формате** и подменяем substring'ом. Это поднимает покрытие дат с ~50% до ~92%.

## [2026-05-21] Boundary-matching критичен для числовых типов

- **Контекст:** первая итерация injection для Type 1 поймала баг — для значения `0.002` функция `f"{value:.2f}"` дала `"0.00"`, эта подстрока нашлась внутри `"0.002"` в ответе, потом replacement сгенерировал тоже `0.00` → self-swap (X→X).
- **Что наблюдали:** 1 self-swap из 758 (≈ 0.1%), но в принципе хрупкий подход к матчингу чисел.
- **Гипотеза почему:** строки `"0.00"`, `"12"`, `"100"` часто являются префиксами/суффиксами других чисел. Substring-поиск без word-boundary матчит «не то».
- **Что делать дальше:** для всех числовых стратегий вводить boundary-check: символ слева/справа не должен быть цифрой или точкой. Также — перебор кандидатов представления значения в порядке убывания специфичности (`str(value)` сначала, форматированные потом).
- **Эффект:** покрытие 73.3% → 69.3%, но 0 self-swap'ов и 0 битых спанов.

## [2026-05-21] In-sample pool swap может создать «двойник» рядом

- **Контекст:** для категориальных строк pool-swap иногда подменяет `jQuery` → `React`, при том что `React` уже стоит рядом в перечислении («React, React»).
- **Что наблюдали:** редкий артефакт, но визуально палится.
- **Гипотеза почему:** in-sample pool по определению содержит соседей того же списка.
- **Что делать дальше:** при выборе альтернативы шарффлить пул и предпочитать те, что не создают дубликата в окне ±4 символа.

## [2026-05-21] Bool с явными rule-based паттернами для топ-полей

- **Контекст:** ранее отказались от bool как «слишком хрупкого». Но 80 из 173 bool-значений приходятся на одно поле `is_correct` (trivia), плюс ещё ~50 на стандартные `isValid`, `exists`, `success`, `stock`, `free_shipping`, `prime_eligible`. Это закрытое множество с предсказуемыми фразами.
- **Что наблюдали:** хардкодом 10 пар (field, regex → replacement) поймали 27 bool-инжекций без единого битого спана.
- **Гипотеза почему:** ассистенты используют стандартные английские формулировки для известных полей; для каждого поля 1-3 варианта покрывают все случаи.
- **Что делать дальше:** добавить ещё 2-3 поля (например, `discount`, `indexed`, `is_anonymous`) если захотим больше bool-кейсов. Сейчас 27 — достаточно.

## [2026-05-21] LLM-fallback на Qwen3-235B покрывает почти все остатки

- **Контекст:** для оставшихся 283 непокрытых rule-based сэмплов прогнали LLM через OpenRouter (модель `qwen/qwen3-235b-a22b-2507`). Промпт просит модель: выбрать в ответе один факт, поддержанный tool output, и предложить substring-level swap с грамматическим контролем word-boundaries.
- **Что наблюдали:**
  - 281 / 283 валидных предложения (**99.3% success rate**).
  - 0 битых спанов, 0 self-swap, 100% reversion на финальном датасете.
  - Параллельный прогон 10 воркерами занял ~2 минуты вместо 15-25 последовательно.
  - Цена: ~$0.30 за все 283 вызова на этой модели.
- **Где LLM остаётся слаба:** ~3 из 20 случаев в выборочной проверке имели грамматический артефакт «has been failed to X» (активный паттерн модели, добавленный негативный пример в промпте не вытесняет его полностью). Это ~15% LLM-инжекций, ~4% от всего датасета.
- **Гипотеза почему:** 235B модель надёжно понимает задачу разметки, JSON-выхода и word-boundary правил; остаются только локальные грамматические зацепки.
- **Что делать дальше:** ничего, для дальнейших задач этого достаточно. Если позже понадобится более жёсткое качество — добавить пост-фильтр по regex `\b(has|have|had|was|were|been|being)\s+\w*\s*failed to\b` (теряем ~10 LLM-кейсов).
